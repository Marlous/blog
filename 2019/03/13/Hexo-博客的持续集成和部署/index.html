Hexo 博客的持续集成和部署

参考：
1、 Hexo的版本控制与持续集成
2、 使用 Travis-CI 持续集成部署 HEXO 博客项目
3、 用 Travis CI 自动部署 hexo

一 概述

  1. 使用：
  • 从零开始，在本地部署环境、安装运行 Hexo 博客，然后将此博客文件夹源码 push 到一个 GitHub 仓库中。
  • 每次写博文直接在对应的 post 文件夹新建博文.md 和与博文同名存图片的文件夹即可。
  • 每次写完博文，直接提交到对应 GitHub 仓库即可（换台电脑就把 GitHub 上对应的源码仓库拉下来）。
  1. 实现原理概述：
  • 第一次在本地安装环境，安装好 Hexo，测试运行博客（从零开始）。
  • 将此博客文件夹的全部内容 push 到一个仓库中（可以忽略 public 文件夹）。
  • 然后使用 CI/CD 服务（相当于在云端布置一个环境）拉此仓库的源码，进行构建。
  • 构建完后推送到 GitHub 上的用来提供 GitHub Pages 博客服务的仓库（之前已建好）中。
  1. 实现原理（使用现成的 CI/CD 服务）：
  • 本地准备好环境和可以运行 Hexo 后，将全部源码推到中间存储用的仓库。
  • 再在 CI/CD 服务器上搭建一个环境，这个环境专门用来生成博客源码。
  • 将本地的源码推到这个云端环境中去，CI/CD 服务器构建完后的博客源码推到博客仓库。
  1. 实现原理（自己用 Docker 搭建 CI/CD 服务）：
  • 直接在远端自己的服务器上搭建一个完整的环境和可以运行 Hexo 后（docker），这个环境专门用来生成博客源码。
  • 只推送自己的博文文件夹到中间存储用的仓库。
  • docker 容器从中间存储用的仓库更新自己本地的 post 文件夹，并生成博客源码。
  • 将构建完后的博客源码推到博客仓库。
  1. 两者区别：
  • 前者相当于完全复制一份本地的博客相关源码到远端（中间存储仓库传送），远端环境中负责构建，然后推送博客源码部署。
  • 后者相当于直接将博客全部源码建在远端，本地只留一个存博文的文件夹，通过中间存储仓库更新远端的 post 文件夹，然后远端构建，再推送博客源码部署。

二 使用现成的 CI/CD 服务

  • 本地已建好可运行的 Hexo 博客。

  • 将此博客文件夹中的部分文件复制到另一个文件夹（my-hexo-cicd），并 push 到自己建立的一个 GitHub 仓库。
